## 第二層｜完整版（`company_context_full`）
**注入對象：僅 daily_plan（每人每日 1 次）與 meeting（晨會每輪發言）**
**Token 預算：約 900–1,100 中文字（= 核心版 + 以下擴充段落）**

### 2.1 客戶組合（全數虛構）

```
【主要客戶】
• 星海航空：本國籍全服務航司，本站最大客戶（日均約 18 架次[FSC-請校對:量級]），
  合約明年到期，續約談判即將啟動，對行李 KPI 盯得很緊。
• 羽澤航空：日系全服務航司，服務手冊極細，每月召開品質檢討會（MQM），
  每季派員來站稽核；溝通需日文書面往返，重視事前報告與鞠躬送機等禮節細節。
• 北風航空：區域航司，冬季班表將新增航班，正待我方報價。
• 藍鵲航空：低成本航司（LCC），過站時間僅 35 分鐘，凡事以價格敏感、
  額外服務逐項計費（ad-hoc price list）為原則。
• 雲嶺物流：貨運包機與郵件處理委託方。
• 機場當局：非航司客戶，是標案（貴賓室經營權、公共區服務委外）的招標方。
```

### 2.2 合約體系（部門的核心工作物）

```
【合約體系】對航司之合約依 IATA SGHA（標準地勤服務合約）架構：
• 主約（Main Agreement）：定義雙方權利義務、責任與賠償上限、終止條款；
  責任條款（Article 8）是法務攻防重點，輕易不動。
• 附錄A（Annex A）：服務項目型錄，逐節勾選本站實際提供之服務
  （客運、機坪、載重平衡與通訊、貨運郵件、支援服務等）。
• 附錄B（Annex B）：本站專屬商務條件——各機型架次費（turnaround fee）、
  額外服務單價表、計價幣別、付款天期、年度調價機制（隨物價/工資指數
  [FSC-請校對:實務調價基準]）、SLA 與服務抵扣（service credit）條款。
• 日常工作八成發生在附錄B：續約談判、增班改價、臨時包機/加班機報價、
  颱風 IROPS 後的臨時人力與延時費用求償結算。
• 版本管理：SGHA 每數年改版，改版年要盤點既有合約是否隨新版轉換[FSC-請校對]。
```

### 2.3 SLA 與 KPI（規範面）

```
【SLA/KPI】常見指標：首件/末件行李送達時間、輪椅等特服旅客銜接率、
報到櫃檯排隊時間、航班過站準點貢獻（地勤可歸責延誤 delay code 歸因）、
行李異常率（每千人次）、地面損傷事件（ground damage）通報時效。
未達標按附錄B的抵扣機制計罰；連續未達標會被客戶要求提改善計畫（CAP）。
部門負責把 KPI 條款談得「可達成、可量測、可申復」，並每月核對站上數據
與航司報表的差異。
```

### 2.4 年度節奏（時間感）

```
【年度節奏】
• 換季（3月底、10月底的 IATA 夏/冬班表）：換季前 6–8 週航司陸續遞交新班表，
  部門要完成增減班的價格確認與附錄B修訂，並會辦營運單位人力班表。
• 旺季：春節（加班機報價高峰）、暑假、連假；颱風季（7–9月）隨時待命處理
  IROPS 費用認定。
• 稽核季：航司年度稽核、ISAGO 相關查核與缺失改善追蹤[FSC-請校對:公司是否持ISAGO]。
• 標案季：機場當局的委外案多集中在特定季度公告[FSC-請校對]。
```

### 2.5 標案實務（招商面）

```
【標案實務】機場當局標案多依政府採購法辦理[FSC-請校對:或促參/設定權利金之商業標]：
公告領標 → 疑義釋疑 → 投標（資格標／技術標「服務建議書」／價格標，
分信封彌封）→ 資格審查 → 評選委員會簡報答詢 → 最有利標評選或開價格標
→ 決標 → 押標金退還、履約保證金繳納 → 簽約。
部門內部流程：投標決策簽核（副總拍板報價底線）→ 服務建議書分工撰寫
（營運數據找站上要、財務數字找財會核）→ 簡報演練 → 送標（紙本裝訂、
騎縫章、截止時間死線）。航司端的 RFP 則較彈性，但常要求兩輪議價。
輸標檢討與得標後的移交建置（人力、制服、系統、通行證）也是部門的事。
```

### 2.6 法遵與安全（紅線）

```
【法遵與安全】公司持航空站地勤業許可，受民航主管機關監理[FSC-請校對:引用條文粒度]；
人員進出機坪管制區需通行證與安全查核；危險品（DG）相關服務需具備資格；
合約中的保險條款（地勤責任險保額）與安全管理系統（SMS）義務不可退讓。
任何對外承諾新服務前，必須先確認營運單位能力與法規許可範圍——
「業務答應了、站上做不到」是部門最忌諱的失誤。
```

### 2.7 部門用語（讓對話像真的）

```
【常用語】「架次」（一起降的服務單位）、「過站」（turnaround）、「站上」（機場現場
營運單位）、「附錄B」（=商務條件的代名詞）、「特服」（輪椅/UM等特殊旅客服務）、
「拉曳/後推」（機坪作業）、「開口項目」（用多少算多少的計次服務）、
「今天幾班？」（問航班量）、「換季包」（換季修約文件一整包）、
「兩信封」（技術標/價格標分封）、「壓線送標」（趕截止時間送投標文件）。
```

## 整合說明（給 AI 開發者：對規格書 v2 的小幅差異，請一併落實）

1. **拆檔**：`prompts/company_context_core.md`、`prompts/company_context_full.md`（full = core + 2.1~2.7）。原規格書中 `{{company_context}}` 變數改為兩個：`{{company_context_core}}`（converse / react / work_progress / relationship 用）與 `{{company_context_full}}`（daily_plan / meeting 用）。
2. **DDL 差異**：`memories.kind` 允許值新增 `'knowledge'`；`seed_world.sh` 依第三層表格的〔發給〕欄，為對應 agent 各寫入一筆 knowledge 記憶（embedding 照常生成；`sim_day=0`、`sim_clock_sec=0`）。
3. **檢索不需改**：knowledge 切片走既有的「近期性×重要性×相關性」檢索自然浮現；因 `last_access` 會更新，常被用到的常識會自然「更容易想起」——這是符合認知架構的副作用，保留即可。
4. **成本估算**：core ≈ 250 字 ≈ 每次呼叫多 ~250–350 tokens；full ≈ 1,100 字，僅每人每日 1 次 daily_plan 與晨會輪次使用。以 9 人/日粗估，比「整份全塞」省 60% 以上背景 token。`[DEFAULT]` 維持此分層；若 FSC 評估真實度不足再上調。
5. **勘誤**：規格書附錄 A.3 第 2 筆 work_item 標題「桃機貴賓室…」請改為「**北原機場**貴賓室經營權投標案—服務建議書」，以符合卷首虛構聲明（不使用真實機場公司名）。
6. **FSC 校對流程**：FSC 修訂本檔後，只需覆蓋 prompts/ 下對應檔案（熱重載生效），knowledge 切片變更則重跑 `seed_world.sh --knowledge-only`（請實作此旗標：只 upsert knowledge 記憶、不動其他資料）。
